Multi-Chain Portfolio Explorer — Flow

What happens from connecting a wallet to seeing a grouped portfolio.

Request flow (happy path)

1
Collect addresses (client)
User connects a wallet (RainbowKit/wagmi) — each connect captures one address — or pastes a read-only "watch" address. Saved to the Zustand store, persisted to localStorage.
↓
2
Fetch portfolio
usePortfolio (TanStack Query) sends POST /api/portfolio with all tracked addresses. Query key includes the address set, so adding/removing a wallet refetches.
↓
3
Validate & fan out (server)
Route Handler validates the body with Zod, then buildPortfolio calls the provider for each wallet inside Promise.allSettled — one wallet/chain failing never sinks the others.
↓
4
Fetch + normalize (provider)
Alchemy tokens/by-address returns balances + metadata + prices for all networks in one call. Each token is normalized: raw balance kept as a BigInt string, and an assetId group key is resolved (registry → native → symbol@decimals).
↓
5
Return flat Position[] + sources
One record per (wallet × token), plus a sources[] array marking each wallet okfailed for the partial-failure UI.
↓
6
Group & render (client)
groupPositions(by) does a pure groupBy — token (default), network, or wallet — summing balances as BigInt and values in USD. Search filters the rows. Switching grouping never refetches.

Token normalization — "symbol alone isn't enough"

USDC on Ethereum and USDC on Base have different contract addresses. We assign both the same assetId (usd-coin) via a curated registry, so they collapse into one row when grouping by token. Native ETH across L2s shares native:eth. Unknown tokens fall back to sym:SYMBOL@decimals (decimals included so unrelated same-symbol tokens don't merge).

Failure & edge states

empty No wallets tracked → prompt to connect / watch.

loading Skeleton rows while the query is in flight.

partial Some wallets fail → banner names them, successful wallets still render.

mock No ALCHEMY_API_KEY → realistic mock data, badge shows "mock".
